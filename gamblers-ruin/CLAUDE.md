# gamblers-ruin -- the Gambler's Ruin MDP viz

A browser-only, no-build, file://-safe educational visualization of the
**Gambler's Ruin** MDP for a **manager audience**. It is one cartridge in the
SML reinforcement-learning gallery (a sibling of `press-your-luck/`,
`last-minute-pricing/`, `pokemon-battle/`, ...). It walks the canonical
**13-scene arc** (title -> tutorial -> playtest -> formalization -> policy ->
trajectory -> return -> Q* -> Bellman -> DP -> DP-caveat -> learning -> recap).

Design source of truth:
`../mdp-gallery/brainstorm/proposals/09-gamblers-problem.md` (the full spec,
exact numbers, scene-by-scene plan) and `../mdp-gallery/reference/pokemon-arc.md`
(the 13-scene DNA). Built by mirroring the multi-file structure, click-step
scene engine, KaTeX vendoring, and precompute approach of `press-your-luck/`
and `last-minute-pricing/`.

## The MDP

- **State** s = your **capital** in whole dollars, s in {0,...,10} (11 rungs of
  a vertical cash ladder). $0 = RUIN (value 0), $10 = GOAL (value 1) are
  terminal; the **9 interior rungs $1..$9** are playable.
- **Action** a = the **stake** you bet: $1 (timid), $2 (medium), $3 (bold),
  clamped to `a <= s` and `a <= 10 - s` (clamped stakes are greyed, never
  chosen).
- **Transition** P = a **biased coin**, win prob `p = 0.4` (drawn with a 40/60
  badge): heads -> s + a, tails -> s - a.
- **Reward** r = **+1 the instant you reach exactly $10**, 0 otherwise; gamma =
  1. So every V*(s) IS a win-probability in [0,1], the most manager-legible
  number possible.
- **The twist (state-dependent optimal action):** with the unfavorable coin the
  optimal stake is **non-monotone**: timid at the edges, BOLD through the
  dangerous middle. Optimal stakes $1..$9 = **$1, $2, $3, $3, $3, $3, $3, $2,
  $1** (the $5 rung is a genuine $2==$3 tie, broken to the larger stake). Flip
  the coin favorable (p >= 0.55) and the whole pattern collapses to "always
  $1" (proven in the precompute).

### Engine conventions (shared with the gallery)

- `window.Gambler` is aliased to `window.Battle`, and `window.Stakes` to
  `window.Moves`, so the reused `bellman.js` / `sarsa.js` consume them unchanged.
- The Q-table is a clean 9x3 (`Q[stateIndex*3 + stakeIdx]`, stateIndex =
  capital-1). **Clamped stakes** carry `null` (precompute) / `-Infinity`
  (runtime) so they are never argmax and render disabled.
- Scenes return `{ onEnter?, onLeave?, onNextKey?, onPrevKey? }`; returning
  `true` from a key handler consumes the keystroke for an internal step.
- `#scene=N` deep-links; `&instant` disables fade; `&run` auto-triggers a
  scene's primary animation for headless capture; `&theme=light|dark`.

## The rigor gate (precompute)

`node precompute/build-datasets.js` loads the verified runtime engine via a
`window` shim (single source of truth), runs value iteration, trains a
model-free learner, **asserts**, and only then writes `data/datasets.js`. If any
assertion fails the file is NOT written. Verified assertions:

1. VI converges (maxDelta < 1e-9).
2. **Optimal policy == [1,2,3,3,3,3,3,2,1]** exactly ($5 tie -> larger stake).
3. **V* matches the spec to 3 decimals**: 0.039, 0.096, 0.166, 0.241, 0.318,
   0.416, 0.545, 0.649, 0.790.
4. The **only** exact Q* tie among legal stakes is at $5 (bet$2 == bet$3), so a
   snapshot diff does not flap.
5. Bold $3 strictly beats timid $1 across the free-choice middle $3..$7.
6. Two hand backups reproduce V*: Q*($1,$1)=0.4*V*($2); Q*($9,$1)=0.4+0.6*V*($8).
7. The model-free learner reproduces the DP optimal stake on **all 9 rungs**
   ($5 tie accepted), and its greedy win-rate from $5 lands within 0.02 of
   V*($5).
8. At favorable odds p=0.55 the optimal policy collapses to all bet $1.

## SARSA vs Q-learning -- an important, deliberate choice

The proposal asks for "SARSA whose learned policy converges to that table." It
does **not**. This was established empirically (see the tuning done while
building): on this rigged-coin ladder the Q-gaps between stakes are tiny (e.g.
$3: 0.154 vs 0.166) and **on-policy SARSA learns the value of the eps-soft
policy it actually follows, which genuinely favors TIMID bets** (a stray
exploratory bet derails a bold run). Pure SARSA converges to a timid-leaning
policy (~5/9 agreement, robust across seeds and schedules; even GLIE annealing
and ensembling do not recover the bold middle -- averaging converges to the
*wrong* timid fixed point because the bias is systematic, not zero-mean).

The bold-middle table is recovered by **off-policy Q-learning** (bootstrap on
the best next stake), which learns the value of the optimal greedy policy
regardless of exploration. With a **Robbins-Monro decaying step size**
(alpha = 1/(1+visits)^0.75) its estimates converge to true Q* (so the sub-0.02
gaps resolve), recovering all 9 rungs across every seed.

So the model-free learner (precompute Phase 2 + the live scene 11) is
**Q-learning**, framed honestly as model-free TD control ("learn Q by playing,
flip-outcome-adjust, with epsilon to keep exploring"). Scene 11 carries a
one-line note on the on-policy/off-policy distinction (cautious vs optimal),
which is itself a richer teaching point. Both updates live in `sarsa.js`
(`update` = on-policy SARSA, `qLearningUpdate` = off-policy) so a future variant
can show the contrast live. **This is the one substantive deviation from the
literal spec; it is documented in the build report and is the truthful choice.**

## Regenerating the data

```
node precompute/build-datasets.js
```
Re-run after any change to `stakes.js` / `gambler.js` / `bellman.js` /
`sarsa.js`. Payload is ~16 KB (V*, Q*, policy, per-sweep DP fill frames, 10
SARSA snapshots, the win-rate curve, return bars, recap copy, KaTeX strings).

## QA (this Mac hangs headless Chrome)

Use the watchdog recipe (launch detached with a fresh `--user-data-dir`, poll
for the PNG, kill the process). Shots go in `.shots_tmp/` (gitignored).
**Mobile gotcha:** old headless Chrome with `--window-size=390,844` does NOT
emulate a mobile layout viewport (it renders at desktop layout width and shows
the left slice, which looks like overflow). To verify true 390px responsive
layout, render the page inside a `width:390px` `<iframe>` and screenshot that.
The responsive layout was verified clean this way at 390px for all 13 scenes.

## House-style note (deviation worth knowing)

Requirement #3 specified the "light editorial theme (cream #f9f7f1, ink
#1a1a1a, serif headings)" and this viz implements exactly that. The two named
exemplars (`press-your-luck/`, `last-minute-pricing/`) actually ship the
gallery's **Press Start 2P pixel** theme, not the editorial one -- so this
cartridge looks different from those two siblings. It matches the editorial
spec text and the global "light theme + standard fonts" preference; if arcade
consistency with the pixel siblings is wanted instead, the theme is isolated to
`css/style.css` tokens + a few widget classes and can be reskinned.
